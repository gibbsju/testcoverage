Context and Methodology

Target setup: Tart VMs on Mac Minis (16GB RAM, 2 VMs per host = ~8GB/VM, macOS licensing limit).

Classification approach:

  • High: Suite runs as virtualization: virtual on Linux/Windows (default), uses default instance size, no GPU/video/heavy-CPU indicators in name or config.
  • Medium: Suite is virtual on other platforms but has graphics (reftest, WebGL, WebGPU, canvas, screenshots), media (video codecs, encrypted media), or loopback video dependencies that need verification on macOS VMs.
  • Low: Suite explicitly requires highcpu or large instance sizes, or needs both loopback video and hardware GL — likely exceeds 8GB VM capacity.
  • Performance: All talos, browsertime, and awsy suites — require bare metal for stable, reproducible performance numbers.

Key finding: macOS CI config has no virtualization branching — all tests go to physical Mac pools regardless. The virtualization field only affects Windows worker selection. This analysis uses the Linux/Windows virtual status as a proxy for "doesn't need bare metal."

Architecture note: Both Intel (Mac Mini) and Apple Silicon (M4 Mac Mini) hardware is in use. 115 tests only have coverage on Intel, 201 only on Apple Silicon. Neither architecture can be fully dropped without coverage loss.
